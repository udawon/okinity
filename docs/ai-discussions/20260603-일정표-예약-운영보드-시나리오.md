# 일정표 · 예약 운영 보드 — 상황별 시나리오

작성일: 2026-06-03 · 대상: OKINITY 예약/일정 시스템 재설계

## 0. 설계 원칙 (인터뷰 결과 반영)

- **동시진행 고정 한계 없음** → 시스템은 자동 차단하지 않고 *정보 제공 + 사람 판단 보조*
- **투어 종류마다 시간 단위 다름** → 고정 격자 ❌, 자유 타임라인(분 단위 배치)
- **마감도 '문의 가능'** → 휴무만 차단. "마감"은 *"예약 많음 · 조율 문의"* 로
- **운영자: 건별 카드 + 시간 이동·재배치** → 실제 예약(inquiries)을 드래그 카드로

**한 줄 요약**: 일정표는 *막는 게이트*가 아니라 **협상 가능한 정보판**.

## 1. 등장 요소

### 예약 카드 (운영 보드의 단위)
```
┌──────────────────────┐
│ 🟦 펀다이빙            │  ← 대분류 색(스노클링🟩/다이빙🟦/PADI🟪/낚시🟧)
│ 2명 · 가예약           │  ← 인원 · 상태
│ 김다이버 · 010-...    │  ← 고객 · 연락처
│ 희망: 오전            │  ← 방문자 희망 시간대
└──────────────────────┘
```

### 상태 전이
```
신규 ──→ 가예약 ──→ 확정 ──→ 완료
  └────────┴────────┴──→ 취소
```
- **신규**: 방금 들어온 문의(미검토)
- **가예약**: 운영자가 잠정 잡음(조율 중 — 바뀔 수 있음)
- **확정**: 시간·인원 확정, 고객 회신 완료
- **완료**: 투어 종료 / **취소**: 무산

### 운영 보드 (주간 보드 + 하루 상세) ← 확정: 주간 보드 기본

**주간 보드(기본 화면)**: 한 주 7일을 나란히, 각 날짜에 예약 카드 + 조율/혼잡 신호.
```
        6/8  6/9  6/10 6/11 6/12 6/13 6/14
        월   화   수   목   금   토   일(휴무)
오전     ·   🟦   ·    ·   🟩   🟦    —
오후     ·    ·   ·    ·    ·   🟩7  ⚠ 조율!   ← 6/14 겹침 신호
```

**하루 상세(날짜 펼침)**: 세로 타임라인에서 카드 시각을 드래그/편집(다이빙 10:30→13:00).
```
6/14(토) 상세                        [+ 직접 예약 추가]
──────────────────────────────────────────────
10:30 ┆ 🟦 다이빙 2명(가예약)   🟩 스노클링 7명(신규)
      ┆ 김다이버               박단체(7)  ⚠ 겹침 · 예상 ₩60k vs ₩210k
13:00 ┆ ← 다이빙 카드를 여기로 끌어 이동
──────────────────────────────────────────────
```
예상매출(₩)은 **어드민에서만** 표시(투어별 단가×인원). 고객 비노출.

## 2. 시나리오

### 시나리오 1 — 빈 시간 단순 예약 (해피패스)

**상황**: 6/10 예약 없음. 고객 A가 펀다이빙 2명, 오전 희망.

| 주체 | 동작 |
|---|---|
| 방문자 | 캘린더 6/10(여유) 클릭 → 예약창 → 스쿠버다이빙>펀다이빙 / 오전 / 2명 / 연락처 → 문의 |
| 시스템 | `inquiries`에 **신규** 카드 생성 + 운영자 알림(이메일·카톡) |
| 운영자 | 보드 6/10에 `[펀다이빙·2명·오전·신규]` → **확정** 클릭, 09:00 배정 |
| 결과 | 고객에게 "6/10 09:00 펀다이빙 확정" 회신 |

→ 가장 흔한 흐름. 충돌 없음.

---

### 시나리오 2 — ★핵심★ 다이빙 ↔ 스노클링 조율 (수익 최적화)

**상황**: 6/14 10:30 펀다이빙 2명이 **가예약**(보트·시간 점유). 같은 10:30에 **스노클링 7명 단체**가 신규 문의.

| 주체 | 동작 |
|---|---|
| 방문자(단체) | 6/14 클릭 → **"예약 많음 · 조율 문의"** 표시 + *"이미 예약이 있어 시간 조율이 필요할 수 있어요. 희망 시간대를 남겨주시면 맞춰드릴게요"* 안내 → 그래도 스노클링 7명 / 10:30 희망으로 문의 |
| 시스템 | `inquiries` 신규(스노클링 7명). 보드 6/14 10:30에 **두 카드 겹침** 표시 + 옆에 인원·예상매출 비교(2명 vs 7명) |
| 운영자 | 스노클링이 이득 판단 → 펀다이빙 카드를 **13:00로 드래그**(시간 이동) → 다이빙 고객에게 "오후로 조율 가능할까요?" 연락(카드 메모에 기록) → 스노클링 7명 10:30 **확정** |
| 결과 | 다이빙 고객 동의 시 둘 다 수용(매출 극대화). 거절 시 카드를 원위치로 드래그해 되돌리거나 다른 날 제안 |

→ **시스템은 막지 않고 "겹침 + 가치"를 보여줄 뿐**, 결정·연락은 운영자가. 이게 이 설계의 심장.

### 시나리오 3 — 휴무일

**상황**: 6/21 휴무(운영 안 함).

| 주체 | 동작 |
|---|---|
| 방문자 | 6/21 회색 · 클릭 불가 · "휴무" 표시 → 예약 자체 불가 |
| 운영자 | 일정표(schedule)에서 6/21을 **휴무**로 등록해 둠 |

→ 유일하게 하드 차단되는 케이스.

### 시나리오 4 — 같은 날, 다른 시간 다른 투어 (다회 운영)

**상황**: 6/15 오전 가족 스노클링 4명 확정. 새 고객이 6/15 **오후** 펀다이빙 문의.

| 주체 | 동작 |
|---|---|
| 방문자 | 6/15 클릭(예약 있지만 문의 가능) → 패널에 "이미 예정된 투어: 가족 스노클링" 컨텍스트 → 오후 / 펀다이빙 문의 |
| 운영자 | 보드 6/15에 `오전[스노클링·4명·확정]` + `오후[펀다이빙·신규]` 카드 공존 → 오후 확정 |
| 결과 | 하루 2타임 운영. 충돌 없음(시간 다름) |

### 시나리오 5 — 취소로 자리 회복

**상황**: 6/14 10:30 다이빙 2명 가예약이 고객 사정으로 취소.

| 주체 | 동작 |
|---|---|
| 운영자 | 보드에서 해당 카드 **취소** 처리 |
| 시스템 | 카드 → 취소 상태(흐리게), 보드 혼잡도 갱신(6/14 "여유"로) |
| 효과 | 시나리오 2의 조율 압박이 사라짐 — 스노클링 7명을 10:30 그대로 확정 |

→ 자동 차단이 없으니 "자리 회복"이라기보단 *혼잡도 신호*가 내려갈 뿐.

---

### 시나리오 6 — 당일 워크인 / 전화 예약

**상황**: 당일 아침, 전화로 체험다이빙 3명 즉석 문의(웹 폼 안 거침).

| 주체 | 동작 |
|---|---|
| 운영자 | 보드에서 **[+ 직접 예약 추가]** → 카드 수기 생성(투어·인원·시간) → 확정 |
| 결과 | 웹/전화 예약이 한 보드에서 통합 관리 |

→ 운영자가 직접 카드 생성 가능(웹 폼 외 채널 흡수).

---

### 시나리오 7 — (선택, 4단계) 방문자 혼잡도 힌트

**상황**: 6/14는 예약 밀집, 6/10은 한산.

| 주체 | 동작 |
|---|---|
| 방문자 | 6/14 "예약 많음 · 조율 문의" / 6/10 "여유" 뱃지 — **둘 다 클릭·문의 가능** |
| 시스템 | 예약 카드 밀도로 혼잡도 자동 계산(휴무 제외) |

→ "마감"이라는 막다른 표현 대신 *부드러운 신호*. 문의는 항상 열림.

## 3. 데이터 모델 변경 요약

| 대상 | 변경 |
|---|---|
| `inquiries` | `scheduled_time`(운영자 배정 실제 시각, 방문자 희망 `time`과 별개) 추가 · 상태에 **가예약(tentative)** 추가 |
| 투어 단가 | 세부 투어별 **기준 단가**(어드민 입력, 고객 비노출) → 예상매출=단가×인원 |
| 고객 알림 | 조율 시 **이메일 자동 알림**(이메일 제공 시). 전화·카톡만이면 운영자 수동 |
| `schedule.json`(전시용) | 역할 축소: **휴무 · 공지 · 고정 이벤트**만. 방문자 캘린더는 "휴무 + (선택)혼잡도" |
| 방문자 캘린더 | "마감"→"예약 많음·조율 문의", 휴무 외 절대 차단 안 함 |

### 상태 전이 (확정)
```
신규(고객 신청) → 가예약(운영자 잠정·조율 대상) → 확정(최종) → 완료
        └──────────────┴──────────────┴──→ 취소
```

## 4. 단계별 구현 순서

| 단계 | 내용 | 크기 |
|---|---|---|
| **1** | 방문자: "마감"→"조율 문의", 휴무 외 비차단 + 조율 안내 문구 | 작음(즉시) |
| **2** | 운영 보드 v1: 하루 예약을 카드+타임라인 조회, 상태에 가예약 추가 | 중간 |
| **3** | 운영 보드 v2: 드래그 시간 이동·재배치, 겹침/예상매출 표시 | 큼(dnd) |
| **4** | (선택) 예약 밀도 → 방문자 혼잡도 힌트 | 중간 |

## 5. 확정된 결정 (2026-06-03 인터뷰)

| 항목 | 결정 |
|---|---|
| 예상매출 | **어드민 전용 표시**. 투어별 **기준 단가 입력** → 단가×인원 자동. 고객 비노출 |
| 가예약 단계 | **사용**. 신규 → 가예약 → 확정 → 완료/취소 |
| 고객 알림 | 조율 시 **이메일 자동 알림**(이메일 제공 시) + 운영자 수동 |
| 보드 범위 | **주간 보드** 기본(7일) + 하루 상세 펼침 |

## 6. 남은 설계 메모

- 주간 보드에서 **시각 변경(드래그)**은 하루 상세에서 수행(주간은 날짜 간 이동·개요).
- 이메일 자동 알림은 전화·카톡만 남긴 고객에겐 불가 → 운영자 수동(향후 카카오 알림톡 도입 시 확장).
- 단가는 어드민의 투어 상세(또는 별도 단가 설정)에서 관리.
