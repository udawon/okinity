## 1. 현황 분석

### 1.1 현재 상태
- Next.js 15 App Router + `next-intl`(ko/en/ja, 기본 ko, `localePrefix: always`).
- 메시지 카탈로그 `messages/{ko,en,ja}.json` 은 **106키로 3개국어 완전 정렬**(누락 0).

## 2. 목표

### 2.1 달성 목표
- 어떤 로케일·경로로 진입해도 **약속과 실제가 일치**(언어 전환을 켜놨으면 그 언어로 보인다).
- 브랜드·연락처가 **단일하고 신뢰 가능**.
- 오류 경로(404 등)에서도 **로케일·디자인 일관성** 유지.

### 2.2 성공 기준 (측정 가능)
- en/ja로 홈→투어 상세→예약 폼→메디컬 체크까지 **한국어 잔존 0**.
- 사이트 전 영역 노출 브랜드 명칭이 **OKINITY 단일**, 연락 이메일은 **수신 검증된 주소**.
- `/{locale}/존재하지않는경로` 404가 **헤더·푸터 포함 + 해당 로케일 홈 링크**.
- `npm run typecheck`·`npm run lint`·`npm run build` 무오류, ko 기존 동작 불변.

## 3. 해결 방안 (전부 확정 — 옵션 없음)

### 3.1 [#2] 브랜드/연락처 모순 → OKINITY로 완전 통일

```diff
// messages/{ko,en,ja}.json  (3개 파일 모두)
- "rights": "© {year} PONYOKINAWA. All rights reserved."
+ "rights": "© {year} OKINITY. All rights reserved."
```

```diff
// src/config/site.config.ts
- email: 'hello@ponyokinawa.com',
+ email: 'gapd20@gmail.com',   // 수신 가능한 Gmail(확정). 자체 도메인 확보 시 교체.
```

- **이메일 주소 확정**: `gapd20@gmail.com`(현재 등록·수신 가능). 도메인 미확보 상태에서도 문의 알림이 실제 수신됨. 자체 도메인 확보 시 도메인 메일로 교체(후속).
- 브랜드 표기는 OKINITY 단일. 법인명(PONYOKINAWA) 병기는 하지 않음(향후 법적 요구 발생 시 재검토).

### 3.2 [#3] 최상위 404 로케일·디자인 복구 → catch-all 라우트 + 전역 폴백 보강

원인: 전역 `app/not-found.tsx`(루트 레이아웃 밖)는 로케일 컨텍스트가 없어 한국어·맨몸으로 렌더되고, `/{locale}/오타` 같은 경로가 정상 `[locale]/not-found`로 도달하지 못하고 전역으로 떨어진다.

확정 방안(2중 안전망):
1. **`[locale]` 하위 catch-all 라우트 추가**로 미매칭 경로를 정상 로케일 404로 강제.
```tsx
// src/app/[locale]/[...rest]/page.tsx (신규)
import { notFound } from 'next/navigation';
export default function CatchAll() { notFound(); } // → [locale]/not-found.tsx 렌더(헤더·푸터·3개국어·로케일 홈)
```
   - catch-all은 우선순위 최저라 기존 정적/동적 라우트를 침범하지 않음. `/admin`은 `[locale]` 밖이라 무영향.
2. **전역 폴백(`app/not-found.tsx`) 보강** — 만약을 위한 최후 폴백도 3개국어 병기 + 링크를 `/`(미들웨어가 로케일로 보냄)로.
```diff
- <p style=...>Page not found</p>
- <Link href="/ko">OKINITY</Link>
+ <p style=...>페이지를 찾을 수 없습니다 / Page not found / ページが見つかりません</p>
+ <Link href="/">OKINITY</Link>
```

### 3.3 [#4] 투어 상세 뒤로가기 일관화 → 홈 앵커로 통일

```diff
// src/app/[locale]/tours/[slug]/page.tsx
- const hubHref = entry.categoryId === 'diving' || entry.categoryId === 'padi'
-   ? `/${entry.categoryId}` : '/';
+ const hubHref = '/#activities';   // 카테고리 허브 부재 → 모든 투어가 홈 액티비티 섹션으로 일관 복귀
```
- 카테고리 허브 페이지(`/diving`·`/padi`)는 이미 대표 투어로 redirect되는 상태라, 뒤로가기를 그쪽으로 보내면 "목록"이 아니라 다른 상세로 착지하는 모순이 발생. 모든 투어를 홈의 액티비티 섹션 앵커로 통일.

### 3.4 [#5] 페이지 타이틀 정비
- 하드코딩 한국어 title을 `next-intl` 키로 치환: reserve→`예약/Reservation/予約`(신규 키), about→**기존 `nav.about` 재사용**(소개/About/紹介).
- `tours/[slug]` 미존재 시 fallback title을 조정해 **`OKINITY | OKINITY` 중복 제거**(투어명 없으면 빈 title → 템플릿이 `OKINITY` 단독 출력).
- (홈 기본 타이틀은 이미 로케일별로 정상 → 변경 불필요.)

### 3.5 [#1] i18n 본문 전면 이관 (전면 이관 우선 — 임시책 없음)

> 결정: en/ja 방문자에게 임시 배너/언어버튼 숨김 없이, **한 번에 전체 번역을 완료**해 공개. 미완 노출 구간을 두지 않는다.

**(a) 고정 카피 — Claude가 직접 en/ja 작성해 코드에 반영**
대상 컴포넌트의 한국어를 `next-intl` 키로 이관하고, `messages/{en,ja}.json`에 번역을 채운다.
- `OceanHome.tsx` + `ocean-home-data.ts` (Hero·카드 4종·통계·신뢰·후기·섹션 카피)
- `ReservationForm.tsx` (폼 라벨·옵션·버튼·안내·성공/실패 메시지)
- `MedicalCheckModal.tsx` (메디컬 체크 전 문항·버튼)
- `ScheduleCalendar.tsx` / `ReservePlanner.tsx` (월 이동 aria·"+예약"·"🚫 휴무"·범례 안내·"날짜를 선택하세요" 등)
- `tours/[slug]/page.tsx` (뒤로가기·"포함 사항"·미공개 시 "상세 준비 중" 안내·예약 CTA)
- `about/page.tsx` (metadata 포함)
- `reserve/page.tsx` 상단 h1·안내 문구
- 레거시 `Hero.tsx`: 라이브 미사용 → **삭제(또는 명시적 보관)**. 기존 `hero` 네임스페이스 카피는 현재 화면 문구와 다르므로 OceanHome 실제 카피 기준으로 키를 새로 정리.

**구현 디테일(1차 검토 자동 결정 반영)**
- **운영자 데이터 무결성 — 문의 `product`는 한국어 고정**: `ReservationForm`이 API로 보내는 `product`(투어 종류·세부 프로그램)는 **운영자(어드민/이메일 알림)가 읽는 값**이므로 방문자 언어로 번역하면 안 된다. 따라서 `ACTIVITIES`의 한국어 `title`·`tours[].name`을 **운영자용 표준(canonical) 라벨로 유지**하고, 화면 표시(`<option>` 텍스트)만 번역한다. `<select>`의 `value`는 기존대로 id/slug, 제출 시 `product`는 한국어 canonical에서 조립. (근거: 동적 콘텐츠=운영자/한국어 단일 출처 원칙, 변경 영향 최소화)
- **표시 라벨은 기존 `nav` 키 재사용**: 예약 폼 대분류/세부 프로그램 표시명은 이미 3개국어로 존재하는 `nav.{snorkeling,diving,padi,fishing}` 및 `nav.tours.*`(id·slug 매핑)를 재사용해 중복 번역을 피한다.
- **배열형 카피 번역 방식**: OceanHome의 카드 4종·통계·신뢰·후기 등 배열 데이터는 신규 `home` 네임스페이스에 **구조화 키**(예: `home.activities.snorkeling.title`)로 두고, 후기처럼 순수 배열은 `t.raw()`로 읽는다.

**구현 디테일(2차 검토 자동 결정 — 저위험 설계로 정정)**
- **`ocean-home-data.ts` 텍스트는 제거하지 않고 canonical로 유지**(1차의 "비텍스트만 유지"는 타입 파급이 큼 — `tour.ts` `TOUR_CATALOG`, `tours/[slug]`의 `entry.name`/`categoryTitle`, `product` 조립이 모두 의존). 표시는 **i18n 오버레이**로만 처리:
  - OceanHome 카드/통계/신뢰/후기 표시 = `home` 네임스페이스. ko 값은 현재 한국어와 **1:1 동일**(ko 화면 불변).
  - 투어명/카테고리명 표시(예약 폼 옵션, `tours/[slug]` 헤더) = 기존 `nav.tours.*`·`nav.<category>` 재사용(id·slug 매핑). 매핑 미존재 시 ocean-home-data 한국어로 폴백.
  - `product`(운영자)·카탈로그·라우팅 = ocean-home-data 한국어 canonical 그대로.
- **키 정합**: 신규 `home` 및 reserve 타이틀 키는 **ko/en/ja 3개 파일에 동일 구조로** 추가(누락 시 next-intl 키 노출). ko 신규 값은 현재 하드코딩 문구와 글자 단위 일치.
- **캘린더 라벨 일관화**: `ScheduleCalendar.eventLabel()`이 하드코딩한 "오전만 가능/오후만 가능" 대신, 이미 서버에서 전달되는 `statusLabel`(schedule 네임스페이스·3개국어)을 사용. "🚫 휴무"·"+예약"·범례 안내문도 동일 패턴으로 키화.

**(a-1) 번역 카피 가이드라인 (3차 검토 — 직역 금지 문구)**
- 한국어 특화 문구는 로케일에 맞게 **의역**: "24시간 안에 한국어로 맞춤 일정" → en "We'll reply within 24h with a tailored plan", ja "24時間以内にプランをご案内". ("한국어로"는 ko 한정.)
- 셀링포인트 "한국어 가이드 동행"·"한국어로 편하게"(ASSURANCES) → en/ja는 "guidance in your language"/「日本語対応」류로 적응. (방문자 언어로 응대한다는 약속과 일치.)
- 후기(TESTIMONIALS): 한국인 이름·도시(김지은·서울)는 유지, **인용문·맥락만 번역**.
- 메디컬 체크 등 안전 문구는 의미 정확성 우선(검수 필수, 7.1).

**(b) 어드민 동적 콘텐츠 — 현재 내용만 번역, 어드민 언어탭은 후속**
DB(`site_content`)에 한국어로 저장된 운영자 입력 글(소개 본문·투어 상세·블로그·공지)은 **현재 들어있는 내용을 Claude가 번역**해 채운다(분량 적음 — 투어 상세 대부분 미공개, 블로그 1건).
- 한계 명시: 어드민에 언어별 입력 칸이 없어, **향후 운영자가 새로 쓰는 글은 한국어로만** 입력됨. 어드민 다국어 입력(언어탭) 기능은 **별도 후속 과제**로 분리.
- 단일 출처 원칙: 고정 카피 = `messages/*.json`(i18n 카탈로그), 동적 콘텐츠 = `site_content`(어드민). 이중 관리 금지.

### 3.6 [#6] 저위험 정리
- 캘린더 범례: `available`/`full`도 렌더되므로 범례에 포함하거나, 비노출 의도면 렌더에서도 제외(코드-주석 의도 일치).
- 예약 폼 인원 입력에 `max={50}` 추가(서버 검증과 일치) + 초과 시 안내.
- `Footer` 로고 `next/image`에 `priority` 추가(LCP 경고 해소).
